Top-Down Racing Game in C# – Detailed Development Plan

=== 1. Project Overview & Goals ===
- Goal: Build a fully functional, standalone C# project (no external game engine) that delivers a top‑down racing experience reminiscent of F1/NASCAR.
- Core features:
  * WASD keyboard controls (accelerate, brake, steer left/right)
  * Realistic‑looking cars (F1 or NASCAR sprites)
  * Complete track with pit lane, start/finish line, lap counting
  * Title screen, Settings screen, in‑game HUD
  * Save/Load of progress (current race state, settings)
  * Clean, modular code base that can be compiled with `dotnet build`
- Non‑goals: No 3D graphics, no physics engine, no networking. All systems are hand‑rolled using .NET libraries.

=== 2. High‑Level Architecture ===
The game will be split into clearly separated subsystems.  Each subsystem lives in its own folder and namespace.

- **Engine Core** – Drives the main loop, timing, and orchestrates subsystem updates.
- **Input System** – Captures keyboard state (WASD) using WinForms events.
- **Rendering System** – Draws everything to a double‑buffered bitmap via System.Drawing (GDI+).  The bitmap is blitted to a WinForms Form.
- **Physics System** – Simple kinematic model for cars (speed, acceleration, friction, turning radius).  Handles collisions with track boundaries and pit lane.
- **GameState Management** – Holds the current race state (car positions, laps, pit stop status, timer).
- **UI System** – Title screen, Settings screen, HUD overlay.  UI elements are drawn by the Renderer but receive input via the Input System.
- **Asset Manager** – Loads sprites, track images, configuration JSON files.  Caches resources for reuse.
- **Save/Load System** – Serialises GameState to JSON (`savegame.json`) and deserialises it on request.
- **Configuration** – JSON files for car specs, track geometry, user preferences.

| Namespace | Class | Purpose |
|----------|-------|---------|

| `Assets` | `AssetManager` | Loads PNG/JPEG sprites, track PNG, reads JSON configs. |
| `Physics` | `Car` | Position (`Vector2`), velocity, direction (float in radians), parameters (max speed, acceleration, turn rate). |
| `Physics` | `Track` | Holds track bitmap, collision mask (binary bitmap), pit lane polygon, start line line segment. |
| `Physics` | `CollisionMask` | Helper to test pixel‑perfect collisions using the track mask. |
| `Gameplay` | `LapManager` | Detects crossing of start/finish line, counts laps, stores best lap time. |
| `Gameplay` | `PitStopManager` | Detects entry into pit lane polygon, handles refuel/repair over time. |
| `UI` | `TitleScreen`, `SettingsScreen`, `HUD` | Draw UI elements, respond to menu selections. |

**Utility Types**:
- `Vector2` – Use `System.Numerics.Vector2` for all 2‑D math.
- `RectangleF` / `Polygon` – Simple structs for hit‑boxes.

=== 4. Asset Pipeline ===
1. **Sprites** – 256 × 256 PNG files for each car, layered for different orientations (e.g., 16‑frame sprite sheet covering 360°).
2. **Track Image** – Large PNG (e.g., 2048 × 2048) showing the road, grass, markings.
3. **Collision Mask** – Generated from the track image: white = drivable, black = wall.  Exported as a separate PNG (`track_mask.png`) and loaded as a `Bitmap` for pixel checks.
4. **Configuration JSON** –
   * `cars.json` – List of car definitions (name, maxSpeed, acceleration, turnRate, spriteSheetPath).
   * `track.json` – Path to track image, mask, pit lane polygon points, start line coordinates.
   * `settings.json` – Default resolution, key bindings, audio volume.
5. All assets live under `Assets/` folder; paths are relative to the executable.

=== 5. Rendering Approach ===
- **Framework**: WinForms (`System.Windows.Forms`) with a custom `Form` that owns a `PictureBox` or overrides `OnPaint`.
- **Double Buffer**: Each frame a `Bitmap` the size of the game window is created (or reused) and a `Graphics` object obtained via `Graphics.FromImage`.  All drawing operations happen on this off‑screen bitmap.
- **Sprite Rotation**: Use `Graphics.TranslateTransform` + `RotateTransform` + `DrawImage` to rotate car sprites around their centre.
- **UI Overlay**: After world rendering, draw HUD text (`Graphics.DrawString`) and UI panels.
- **Performance Tips**:
  * Reuse the same `Bitmap` instance; only clear it each frame.
  * Cache `Image` objects in `AssetManager` to avoid repeated loads.
  * Disable anti‑aliasing for pixel‑perfect look, enable for smoother UI.

=== 6. Input Handling ===
- Register `KeyDown` & `KeyUp` events on the main `Form`.
- Maintain a `HashSet<Keys>` of currently pressed keys.
- Translate to logical actions:
  * W → `IsAccelerating`
  * S → `IsBraking`
  * A → `SteerDirection = -1`
  * D → `SteerDirection = +1`
- Provide a `GetAxis()` method returning a float -1..1 for steering.
- Input is read each physics tick (fixed timestep) to guarantee deterministic behaviour.

=== 7. Physics & Car Dynamics ===
**Simplified Kinematic Model** (sufficient for arcade feel):
```
float speed          // current forward speed (m/s)
float acceleration   // constant acceleration when throttle pressed
float deceleration   // natural drag when no throttle
float brakeForce     // stronger negative acceleration when braking
float turnRate       // radians per second at full steering
```
Update per physics tick (`dt`):
1. Apply throttle/brake to modify `speed`.
2. Clamp `speed` to `[0, maxSpeed]` (or allow small reverse values).
3. Apply steering: `direction += turnRate * steeringInput * (speed / maxSpeed) * dt` – steering is proportional to speed.
4. Update position: `position += Vector2.Transform(Vector2.UnitX, Matrix3x2.CreateRotation(direction)) * speed * dt`.
5. Apply friction: `speed *= (1 - frictionCoeff * dt)`.

**Collision**:
- Convert car position to pixel coordinates on the track mask.
- Sample the mask pixel at the car’s centre and at four corner offsets (front‑left, front‑right, rear‑left, rear‑right).
- If any sampled pixel is black (non‑drivable), revert position to previous safe state and optionally apply a small bounce.
- Pit lane detection uses polygon `Contains(point)` test.

=== 8. Game Loop & Timing ===
```
const float FIXED_DT = 1f/60f; // 60 Hz physics
Stopwatch sw = Stopwatch.StartNew();
float accumulator = 0f;
while (running) {
    float frameTime = (float)sw.Elapsed.TotalSeconds;
    sw.Restart();
    accumulator += frameTime;
    while (accumulator >= FIXED_DT) {
        InputHandler.Update();
        Physics.Update(FIXED_DT);
        GameState.Update(FIXED_DT);
        accumulator -= FIXED_DT;
    }
    Renderer.Render(); // renders the latest GameState
    Application.DoEvents(); // pump WinForms messages
}
```
- This pattern guarantees deterministic physics independent of render frame rate.
- `Application.DoEvents()` is safe in a simple WinForms loop; alternatively use a `Timer` but the above gives tighter control.

=== 9. UI Screens ===
**TitleScreen**
- Background image (static track preview or stylised logo).
- Menu options: "Start Race", "Settings", "Exit".
- Navigation via Up/Down arrows or mouse click.

**SettingsScreen**
- Resolution dropdown (windowed sizes).
- Key binding remap (optional, default WASD).
- Audio volume sliders (if audio added later).
- Save button writes `settings.json`.

**HUD (In‑Game Overlay)**
- Speedometer (numeric km/h).
- Lap counter (Current/Total).
- Position (e.g., 3rd/8).
- Pit status indicator.
- Mini‑map (optional – a small scaled view of track with car dot).

All UI elements are drawn after the world via `Renderer.DrawText` and simple shapes.

=== 10. Save/Load System ===
- **GameState DTO** captures:
  * Car positions, velocities, direction.
  * Current lap, lap times.
  * Pit stop progress.
  * Timer elapsed.
- **Serialization**: `System.Text.Json` with `JsonSerializerOptions { WriteIndented = true }`.
- **File Location**: `saves/savegame.json` relative to executable; ensure the `saves` folder exists (create on first save).
- **Loading**: At startup, read `savegame.json` (if present) and reconstruct objects; otherwise start fresh.
- **Settings**: Separate `settings.json` loaded on app start, saved on exit or when changed.

=== 14. Risks & Mitigation Strategies ===
| Risk | Impact | Mitigation |
|------|--------|------------|
| Rendering flicker / low FPS | Poor player experience | Use double buffering, reuse Bitmap, profile with Stopwatch.
| Input lag or missed keys | Controls feel unresponsive | Capture KeyDown/KeyUp at Form level, poll current key set each physics tick.
| Collision false‑positives | Car gets stuck or passes through walls | Use multiple sample points (centre + front corners) and a small tolerance.
| Asset path errors on different machines | Game fails to load | Use relative paths and `Path.Combine(AppContext.BaseDirectory, ...)`.
| Save file corruption | User loses progress | Write to temporary file then replace atomically; validate JSON on load.
| Over‑engineered physics | Too much code for MVP | Start with simple kinematic model; iterate only if needed.
